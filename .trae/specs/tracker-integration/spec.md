# 追踪功能整合到前端 - 产品需求文档

## 概述
- **Summary**: 将基于计算机视觉的位置追踪功能完整整合到普罗米利亚地图工具前端，实现从屏幕/游戏窗口实时抓取小地图，通过 SIFT 特征匹配定位，并将结果实时反馈到网页端地图。
- **Purpose**: 提供实时的游戏内位置追踪功能，让用户在使用地图工具时能够看到自己在游戏世界中的实时位置。
- **Target Users**: 普罗米利亚地图工具的用户，特别是需要实时位置反馈的游戏玩家。

## 目标
- 实现从屏幕或游戏窗口实时截图获取小地图
- 保持现有的 SIFT 特征匹配定位功能
- 确保 WebSocket 实时通信正常工作
- 在前端添加直观的追踪功能控制界面
- 实现追踪状态的实时反馈

## 非目标（范围外）
- 不修改现有的地图标记和路线功能
- 不改变现有的地图切换和管理逻辑
- 不涉及游戏内其他数据的获取
- 不支持多用户同时追踪

## 背景与上下文
- 现有系统是一个纯前端的地图标记工具，使用 Leaflet 作为地图框架
- 已经实现了基本的 WebSocket 集成和追踪标记显示
- 当前追踪程序从视频文件读取小地图，需要改为从屏幕/游戏窗口截图
- 系统使用 Python 进行视觉处理，通过 WebSocket 与前端通信

## 功能需求
- **FR-1**: 屏幕截图功能 - 实现从屏幕或指定游戏窗口实时截图获取小地图
- **FR-2**: 前端追踪控制 - 在网页端添加追踪功能的开启/关闭控件
- **FR-3**: 实时位置反馈 - 将追踪到的位置实时显示在地图上
- **FR-4**: 追踪状态显示 - 在前端显示追踪器的连接状态和工作状态
- **FR-5**: 配置界面 - 提供简单的配置选项，如窗口选择、ROI 调整等

## 非功能需求
- **NFR-1**: 性能 - 追踪过程不应明显影响系统性能，截图和处理频率应适当
- **NFR-2**: 可靠性 - 追踪系统应能稳定运行，即使在游戏窗口大小变化时也能正常工作
- **NFR-3**: 响应性 - 位置更新应足够及时，延迟不超过 500ms
- **NFR-4**: 易用性 - 追踪功能的控制应直观易懂，配置过程简单明了

## 约束
- **技术**: Python 3.7+, OpenCV, WebSockets, Leaflet.js
- **系统**: Windows 操作系统（屏幕截图功能依赖）
- **依赖**: opencv-python, numpy, websockets, pyautogui（用于屏幕截图）

## 假设
- 用户使用 Windows 操作系统
- 游戏窗口在追踪过程中保持打开状态
- 用户拥有足够的系统资源运行追踪程序
- 游戏小地图在屏幕上的位置相对固定

## 验收标准

### AC-1: 屏幕截图功能
- **Given**: 追踪程序已启动，游戏窗口已打开
- **When**: 开启追踪功能
- **Then**: 程序应能从屏幕或指定游戏窗口成功截取小地图
- **Verification**: `programmatic`

### AC-2: 前端追踪控制
- **Given**: 网页端地图工具已打开
- **When**: 用户点击追踪开关
- **Then**: 追踪功能应相应开启或关闭，UI 状态应同步更新
- **Verification**: `human-judgment`

### AC-3: 实时位置反馈
- **Given**: 追踪功能已开启，游戏角色在移动
- **When**: 游戏角色在游戏内移动
- **Then**: 网页端地图上的追踪标记应实时更新位置
- **Verification**: `human-judgment`

### AC-4: 追踪状态显示
- **Given**: 追踪程序运行中
- **When**: 查看网页端追踪状态
- **Then**: 应显示当前连接状态和追踪状态
- **Verification**: `human-judgment`

### AC-5: 配置界面
- **Given**: 追踪程序首次运行
- **When**: 打开配置界面
- **Then**: 应能选择目标窗口并调整小地图 ROI
- **Verification**: `human-judgment`
